## 1. Identificação do tratamento

- **Nome do módulo:** Prontuário fisioterapêutico (avaliação, evolução, plano de tratamento, alta)
- **Sprint que entrega:** [Sprint 20 — Fisio prontuário CID/CIF](../../sprints/20-fisio-prontuario-cid-cif.md)
- **ADR de referência:** ADR 0032 (esperado, Sprint 20) — `signature_policies` por profissional + assinatura ICP opcional fisio
- **Marco regulatório:**
  - **Lei 13.787/2018** (digitalização de prontuário, retenção mínima 20 anos)
  - **COFFITO 414/2012 + 415/2012** (prontuário eletrônico fisioterapeuta — ICP-Brasil opcional se autenticação + audit chain)
  - **LGPD art. 11 II "a"** (tutela da saúde)
- **Finalidade declarada:** registrar avaliação fisioterapêutica + evolução + plano + alta com rastreabilidade completa (autor, timestamps, versões), atendendo COFFITO + LGPD + retenção 20a
- **Categorias de dado tratado:**
  - [x] Saúde (CID, CIF, queixa, exame físico, diagnóstico fisioterapêutico, plano)
  - [x] Composição corporal / antropometria (quando relevante a tratamento)
  - [ ] Genética
  - [ ] Biometria (assinatura biométrica do profissional possível em fase 2)
  - [ ] Wearable (cruzamento com Device Hub é separado, RIPD próprio)

## 2. Base legal (LGPD art. 7º + art. 11)

| Categoria | Base legal | Justificativa |
|---|---|---|
| Saúde clínica | art. 11 II "a" — **tutela da saúde** + COFFITO 414 (obrigação legal do profissional) | Profissional fisio (com CREFITO ativo em `professional_registrations`) executa tratamento sob obrigação legal do conselho |
| Identificação do paciente | art. 7 I (consentimento na adesão) + art. 11 II "f" (proteção da saúde) | Identificação mínima necessária para vincular evolução a paciente |

## 3. Princípios aplicados (skeleton)

- **Finalidade:** registro clínico individual; **não** treina IA sem opt-in explícito (regra 28)
- **Adequação/Necessidade:** dados coletados conforme template COFFITO (avaliação, evolução, plano, alta)
- **Segurança:** RLS multi-tenant + hash chain `audit_log` (regra 39) + assinatura authenticated_lock+audit_chain por default (CREFITO) ou ICP-Brasil opcional (`signature_policies`)
- _Demais princípios serão expandidos no Sprint 20 quando UX + schema estiverem definidos_

## 4. Fluxo de dados (a expandir Sprint 20)

| Camada | Controle | Referência |

### 5.1 Estratégia de retenção 20a (Lei 13.787/2018)

Materialização concreta exigida pelo art. 16 LGPD (retenção legal):

| Camada | Comportamento | Janela |
|---|---|---|
| **Particionamento** | `clinical_records_fisio` declara `@volume_estimate_yearly: 8M` (regra 34) → `PARTITION BY RANGE` em `created_at` (mensal); cada partição com indexes próprios e RLS herdada | Sempre |
| **Quente (PG)** | Partições do **ano corrente + 4 anos** ficam no Postgres (Supabase MVP / Oracle OCI Fase 2). Acesso latência <100ms. | D+0 a D+5a |
| **Cold (R2/S3)** | Partições D+5a+ exportadas em Parquet zstd para Cloudflare R2 (MVP) / AWS S3 us-east-1 (Fase 2 — Object Lock WORM). Job `archive_clinical_records_to_cold()` mensal: dump partition → cifra AES-256 com KMS chave separada → upload R2 → registra em `compliance_retention_log` → drop partition (metadata-only, ms vs hours). | D+5a a D+20a |
| **Restore on-demand** | Se titular ou perícia exigir, runbook `restore-cold-clinical-record.md` (a criar Sprint 20) baixa Parquet + import temporário em `clinical_records_fisio_restored` (RLS reforçada, audit obrigatório). | Qualquer momento ≤D+20a |
| **Drop final** | Drop de partição cold só após D+20a com **dupla confirmação DPO + tenant_owner** (registro em `data_subject_requests` + `audit_log`). Nada é deletado automaticamente. | D+20a+ |
| **Integridade** | Hash chain `audit_log` (regra 39) cobre transições quente → cold → drop. Quebra dispara `system_alerts critical` + bloqueia novos archives até resolução (runbook [`falha-hash-chain.md`](../../runbooks/falha-hash-chain.md)). | Sempre |

**Auditoria interna trimestral** (ADR 0067) inspeciona: (a) latência de archive (deve ficar <30 dias após partição entrar em D+5a); (b) integridade Parquet via re-hash amostral; (c) runs de drop do trimestre vs `data_subject_requests` correspondentes.

**Custo estimado:** com retenção 20a + cold após 5a, custo de storage por tenant médio (~5k pacientes ativos) cai ~80% vs PG quente integral; ainda assim cabe no Cloudflare R2 free tier (10GB) por anos no MVP.

## 6. Avaliação de risco (skeleton)

| Risco | Probabilidade | Impacto | Mitigação |
|---|---|---|---|
| Profissional sem CREFITO ativo assina | baixa | alto | Gate `professional_registrations.situation='ativo'` + verificação periódica (Sprint pós-19) |
| Vazamento de evolução | baixa | alto | RLS + criptografia at-rest + audit |
| Tentativa de edição retroativa | média | alto | Hash chain detecta + UI bloqueia edição pós-assinatura |
| Falha de assinatura ICP (médico cross-fisio) | baixa | médio | Fallback graceful + alerta |
