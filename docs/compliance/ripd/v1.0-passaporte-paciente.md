# RIPD v1.0 — Passaporte do paciente (cross-tenant)

> Relatório de Impacto à Proteção de Dados — exigido por LGPD art. 38 + Resolução ANPD nº 23/2024 para tratamento de larga escala de dados sensíveis com transferência cross-controlador.
> Justifica `patient_data_access_log` (ADR 0077 + regra 42) e o pipeline de vínculo `patient_company_links` + `patient_link_modules`.

- **Versão:** v1.0
- **Data de elaboração:** 2026-04-25
- **Próxima revisão obrigatória:** 2026-10-25 (semestral) ou em mudança substancial do modelo de passaporte
- **Elaborado por:** Fundador (autor do ADR 0077)
- **Parecer DPO:** **Aceito com restrições** — ver seção 8 (LogiFit publica o feature passaporte com revisão jurídica externa adicional antes do primeiro tenant clínico ativar; revisão registrada em `compliance_retention_log`)
- **Hash SHA-256 do conteúdo:** _(será gerado por `scripts/hash-ripd.ts` em CI antes de qualquer merge que toque este arquivo — script criado em Sprint 00 conforme regra 29; campo atualizado automaticamente no commit hook)_

## 1. Identificação do tratamento

- **Nome do módulo:** Passaporte cross-tenant do paciente — vínculo paciente↔empresa com módulos liberados explícitos
- **Sprint que entrega:** [Sprint 02 — CRM unificado (pessoas)](../../sprints/02-geral-crm-pessoas.md)
- **ADR de referência:** [ADR 0077](../../decisions/0077-passaporte-paciente-vinculo-cross-tenant.md)
- **Regra dura:** [regra 42](../../rules.md) (rules.md)
- **Finalidade declarada:** permitir que paciente leve seu histórico de saúde **resumido** (não bruto) entre tenants distintos do LogiFit (ex: Academia → Fisio → Nutri) com consentimento granular por módulo, sob auditoria 5 anos
- **Categorias de dado tratado (que cruzam tenant):**
  - [x] Saúde (resumo de prontuário — nunca SOAP completo)
  - [x] Composição corporal / antropometria (séries de medições)
  - [x] Frequência de treino / dieta (agregados)
  - [x] Wearable agregado (não raw)
  - [ ] Genética (não aplicável)
  - [ ] Biometria (não cruza)
  - [ ] Financeiro (NUNCA cruza — limite duro)
  - [ ] Notas privadas profissional / hipóteses diagnósticas (Nível 5 NUNCA cruza)
  - [ ] Prontuário CFM original (NUNCA cruza — só resumo paciente)

## 2. Base legal (LGPD art. 7º + art. 11)

| Categoria | Base legal específica | Justificativa |
|---|---|---|
| Saúde resumida cross-tenant | **art. 11 II "a" — tutela da saúde** + **consentimento explícito do titular** (LGPD art. 7º I) materializado em `patient_link_modules` ativos com `accepted_at` | Profissional executor no tenant destino atua sob tutela direta; paciente consente módulo a módulo; substituição automática (novo vínculo do mesmo módulo desativa anterior) é confirmada pelo paciente |
| Antropometria cross-tenant | art. 11 II "a" + consentimento | Mesmo enquadramento; nível "Antropometria" é o 2º dos 5 níveis de dados do ADR 0077 |
| Wearable cross-tenant | art. 11 II "d" — execução de políticas públicas (saúde) + **consentimento específico por provider** (já em [ADR 0049](../../decisions/0049-device-hub-wearables-clinicos.md)) | Cross-tenant de device readings só com consent específico do provider + módulo ativo |

**Regra dura:** sem `patient_link_modules` ativo + `data_level_max` cobrindo a categoria, sistema bloqueia leitura cross-tenant via função SQL `has_cross_tenant_access()` + lint `cross-tenant-read-must-log`.

## 3. Princípios aplicados (LGPD art. 6º)

- **Finalidade:** continuidade clínica entre profissionais que tratam o mesmo paciente em tenants distintos
- **Adequação:** dados cruzam **resumidos** (ex: "lesão lombar ativa, restrição: sem deadlift"), não brutos (SOAP completo)
- **Necessidade:** 5 níveis de dados (Identidade → Antropometria → Treino → Clínico → Workspace); `data_level_max` por vínculo limita o que cruza; Nível 5 nunca cruza
- **Livre acesso:** paciente vê em `/meu/privacidade/compartilhamento` quem está vinculado + módulos liberados + histórico de acessos
- **Qualidade:** dado original permanece no tenant origem; cross-tenant entrega snapshot computado on-read (sempre atual)
- **Transparência:** UX de aceite de invite mostra explicitamente quais módulos + nível de dados serão liberados antes do paciente confirmar
- **Segurança:** RLS multi-tenant + função `has_cross_tenant_access()` + audit `patient_data_access_log` síncrono não-bloqueante + criptografia at-rest (ADR 0073 camada 4)
- **Prevenção:** lint `cross-tenant-read-must-log` em CI bloqueia commit que lê cross-tenant sem log; constraint global "1 módulo ativo por paciente em toda a rede" evita duplicação
- **Não discriminação:** profissional do tenant destino vê apenas o que é necessário para tratamento; financeiro do tenant origem NUNCA cruza
- **Responsabilização:** `patient_data_access_log` particionado mensal, retenção 5 anos (ADR 0072), com `reader_user_id`, `reader_tenant_id`, `source_tenant_id`, `module_type`, `category`, `request_id` para responsabilização individual

## 4. Fluxo de dados

```
[Profissional tenant B] solicita vínculo → invite WhatsApp/email → [Paciente]
                                                                       ↓
                                             [Paciente cria conta OU loga]
                                                                       ↓
                                  Tela de pedido pendente: aceitar parcial ou total
                                                                       ↓
                          patient_company_links + patient_link_modules ativos
                                                                       ↓
[Profissional tenant B] lê dado cross-tenant → has_cross_tenant_access() valida →
   leitura permitida + INSERT patient_data_access_log síncrono não-bloqueante →
   resumo entregue (não bruto)
```

- **Origem:** paciente (consente) + profissional (solicita)
- **Quem acessa cross-tenant:** roles clínicas do tenant destino (`medico`, `fisio`, `nutri`, `personal`, `enfermeiro`) + `gerente_filial` quando aplicável
- **Cross-border?** Não — Postgres LogiFit no Brasil (Supabase MVP em SP / Oracle Cloud SP pós-19b); IA pode usar Vertex AI SP (BR) por default ([ADR 0064](../../decisions/0064-ia-arquitetura-gemini-default-byok-rag.md))
- **Retenção:** vínculo persiste enquanto não revogado; access log 5 anos (ADR 0072)
- **Anonimização/eliminação:** paciente revoga vínculo em `/meu/privacidade/compartilhamento` → cessam novas leituras; access log mantido para auditoria (não-deletável)

## 5. Medidas de segurança

| Camada | Controle | Referência |
|---|---|---|
| Isolamento | RLS multi-tenant + `has_cross_tenant_access()` SQL | regra 1, 42 |
| Criptografia at-rest | AES-256-GCM + KEK por tenant | ADR 0073 camada 4 |
| Criptografia in-transit | TLS 1.3, HSTS preload | regra 35 |
| Auditoria síncrona | `patient_data_access_log` particionado mensal, retenção 5a, hash chain via audit_log central | regras 5, 39, 42 |
| Lint CI | `cross-tenant-read-must-log` bloqueia commit sem log | regra 42 |
| Constraint global | 1 módulo ativo por paciente — substituição com confirmação | ADR 0077 |
| Anti-spam invite | rate limit 50 invites/dia/tenant + 3 invites/CPF/30d | Sprint 02 |
| Limites duros não-cruzáveis | financeiro / Nível 5 / prontuário CFM original / dado de outras pessoas | ADR 0077 |

## 6. Avaliação de risco

| Risco | Probabilidade | Impacto | Mitigação |
|---|---|---|---|
| Profissional do tenant destino abusa de leitura cross-tenant | média | médio | Audit log 5a + alerta em volume anormal (>50 leituras/h) + retaliação contratual |
| Paciente aceita vínculo sem entender o que libera | alta | médio | UX explícita por módulo + nível de dados antes de confirmar; tela `/meu/privacidade/compartilhamento` permite revogar a qualquer momento |
| Substituição automática de vínculo confunde paciente | média | baixo | Confirmação obrigatória na UX antes de desativar vínculo anterior |
| Data leak via lint bypass | baixa | alto | Lint custom + revisão de PR + audit captura mesmo se commit passa |
| Tenant origem não autorizado a "espionar" leituras do destino | baixa | médio | Cross-tenant é unidirecional consciente; tenant origem NÃO vê quem leu o que (só paciente vê em `/meu/privacidade/acessos`) |

## 7. Direitos do titular implementados

- [x] **Confirmação do tratamento** — `/meu/privacidade/compartilhamento` lista todos os tenants vinculados + módulos
- [x] **Acesso ao histórico de leituras** — `/meu/privacidade/acessos` lista quem leu o que + quando + por quê (paginado, particionado mensal)
- [x] **Correção** — não aplicável (acesso log é append-only)
- [x] **Anonimização/bloqueio/eliminação** — revogar vínculo em `/meu/privacidade/compartilhamento` cessa novas leituras; histórico de acessos mantido por obrigação de retenção
- [x] **Portabilidade** — export JSON de vínculos + access log em `/meu/dados/export`
- [x] **Revogação de consent** — por módulo individual ou vínculo inteiro; efeito imediato
- [x] **Informação sobre compartilhamento** — UX nativa do passaporte
- [x] **Revisão de decisão automatizada** — substituição automática de vínculo é confirmada pelo paciente (não é decisão automatizada não-revisável)

SLA atendimento: **15 dias** (LGPD).

## 8. Restrições do parecer DPO

DPO aceita o tratamento com as seguintes condições obrigatórias antes da ativação em produção:

1. **Revisão jurídica externa** (escritório especializado em LGPD-saúde) confirmando enquadramento de "consentimento explícito + tutela da saúde" — registrada em `compliance_retention_log`
2. **Primeiro tenant clínico (médico/fisio com CFM/COFFITO)** ativa passaporte apenas após **30 dias de operação MVP estável** sem incidentes (LGPD art. 50 — princípio da segurança)
3. **Auditoria interna trimestral** desta tabela `patient_data_access_log` com amostragem de 1% das leituras + revisão de motivos
4. **Rate limit ajustável** se sinais de abuso emergirem (default 50 invites/dia/tenant pode cair pra 20)
