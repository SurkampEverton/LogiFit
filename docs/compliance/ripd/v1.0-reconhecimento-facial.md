## 1. Identificação do tratamento

- **Nome do módulo:** Check-in por reconhecimento facial em academia (face embedding gerado on-device ou no servidor; comparação 1:N para identificar member entrando) — modalidade `facial` em `access_devices.auth_modes`
- **Sprint que entrega:** [Sprint 08 — Academia controle de acesso](../../sprints/08-academia-controle-acesso.md) — schema `member_face_embeddings` previsto
- **ADR de referência:** ADR 0018 (esperado Sprint 08) — Hardware da catraca + modalidade de auth
- **Marco regulatório:**
  - **LGPD art. 11** (biometria é dado pessoal sensível)
  - **Resolução ANPD nº 4/2025** (biometria — guia orientativo)
  - **Princípio da finalidade** (art. 6 I) — embedding **não pode** ser usado para autenticação financeira ou outras finalidades sem novo consent
- **Finalidade declarada:** identificar membro ativo na entrada da academia (substituindo catraca por TAG/cartão); **não** autoriza pagamento, **não** alimenta IA, **não** é compartilhado cross-tenant
- **Categorias de dado tratado:**
  - [x] Biometria (face embedding 512-d float vector + foto de referência opcional)
  - [x] Identificação (vínculo embedding ↔ member_id)
  - [ ] Saúde
  - [ ] Genética

## 2. Base legal

| Categoria | Base legal | Justificativa |
|---|---|---|
| Face embedding | **art. 11 I — consentimento específico e destacado para biometria** | Paciente recebe tela dedicada com explicação clara antes do enrollment; aceite registrado em `compliance_retention_log` com hash do termo + timestamp |
| Foto de referência (opcional) | art. 7 I + art. 11 I | Para reonboarding em caso de perda do embedding |

**Alternativa obrigatória:** member que não consente recebe **TAG/cartão tradicional** sem prejuízo de plano ou tratamento (princípio LGPD da não-discriminação).

## 3. Princípios (skeleton)

- **Finalidade restrita:** embedding usado **somente** para check-in; nunca para autenticação financeira, IA, marketing
- **Adequação:** modelo treinado para verificação 1:1 / identificação 1:N em pequena escala (até 5.000 members por unit)
- **Necessidade:** apenas embedding de baixa dimensão (512 floats); foto de referência cifrada com KEK por tenant
- **Segurança:** embedding nunca sai do tenant; RLS por unit; criptografia at-rest por campo; deleção imediata em revogação

## 4. Fluxo de dados (a expandir Sprint 30)

```
Member em onboarding → tela consent biometria explícita →
  enrollment: foto via webcam → face embedding (FaceNet/InsightFace on-prem) →
    INSERT face_embeddings (RLS por tenant_id, KEK por tenant) →
      INSERT compliance_retention_log (consent_hash, term_version)

Check-in: câmera unit → embedding → comparação cosine 1:N
  no scope (tenant_id, unit_id) → match (>0.85) → INSERT attendances
  → audit_log
```

## 5. Medidas de segurança

| Camada | Controle | Referência |
|---|---|---|
| Consent reforçado | tela dedicada + termo versionado + opt-out 1-clique em `/meu/privacidade` | LGPD art. 11 I |
| Criptografia at-rest | AES-256-GCM por campo + KEK por tenant | ADR 0073 camada 4 |
| Isolamento | RLS por (tenant_id, unit_id); embedding nunca cruza tenant | regras 1, 42 |
| MFA do operador | role `recepcao` MFA opcional **mas** acesso ao tenant admin de biometria exige MFA recente | regra 43 |
| Retenção | embedding deletado em **D+90** após revogação OU saída do member; foto referência D+30 | regra 34 |
| Auditoria | toda leitura/comparação grava `audit_log` | regra 39 |
| Anti-spoofing | liveness detection (piscar olhos) — fase 2; MVP pode usar foto estática + alerta UX |

## 6. Avaliação de risco

| Risco | Probabilidade | Impacto | Mitigação |
|---|---|---|---|
| Vazamento de embeddings (irreversível — biometria não rotacionável) | baixa | **crítico** | KEK por tenant + criptografia campo + tabela isolada com permissões mínimas |
| Match incorreto (false positive) permite entrada de não-member | baixa | médio | Threshold 0.85 + alerta operacional + log de fallback manual |
| Discriminação por viés do modelo (raça, idade) | média | alto | Modelo escolhido com auditoria de fairness publicada + fallback TAG/cartão garantido |
| Coleta sem consent explícito (operador "esqueceu" de mostrar termo) | média | crítico | Gate Server Action exige `consent_hash` válido antes de criar embedding; sem consent → bloqueia |
| Uso secundário (autoridade pública pede acesso) | baixa | crítico | Sub-processadores não recebem embedding; LogiFit DPO recebe ordem judicial e responde — embedding não é entregue cifrado fora do tenant |

## 7. Direitos do titular

- **Revogação 1-clique** em `/meu/privacidade/biometria` → embedding deletado em D+0 (não há retenção legal além disso); fallback TAG/cartão imediato
- Confirmação visual do embedding existente (não a foto, apenas "ativo desde DD/MM")
- SLA **15 dias úteis** mas revogação biométrica **D+0 obrigatório**
