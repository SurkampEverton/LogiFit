## 1. Identificação do tratamento

- **Nome do módulo:** Device Hub — integração com wearables clínicos (Apple Watch, Garmin, Fitbit, Polar, Oura, Whoop, Withings) coletando HR, VFC, sono, passos, calorias estimadas, eventos cardíacos
- **Sprint que entrega:** [Sprint 32 — Device Hub](../../sprints/32-geral-device-hub.md)
- **ADR de referência:** [ADR 0049 — Device Hub wearables clínicos](../../decisions/0049-device-hub-wearables-clinicos.md)
- **Marco regulatório:**
  - **LGPD art. 11 II "a"** (tutela da saúde — HR/VFC podem indicar condição cardiovascular)
  - **CFM 2.454/2026** se interpretação clínica automatizada → SaMD II (alertas cross-module são SaMD II — ver [samd-classification.md](../samd-classification.md))
  - **Termos do provider** (Apple HealthKit, Google Fit, Garmin Health, etc.) — DPA específico por provider
- **Finalidade declarada:** alimentar acompanhamento clínico (fisio/nutri) e operacional (academia) com dados objetivos do dia-a-dia do paciente; **não** substitui exame médico
- **Categorias de dado tratado:**
  - [x] Saúde (HR, VFC, eventos cardíacos quando reportados)
  - [x] Wearable raw (séries minuto-a-minuto)
  - [x] Wearable agregado (médias diárias)
  - [ ] Genética
  - [ ] Biometria (foto/face — não é wearable)

## 2. Base legal

| Categoria | Base legal | Justificativa |
|---|---|---|
| HR / VFC / sono | art. 11 II "a" + **consent específico por provider** (regra ADR 0049) | Paciente conecta provider em `/app/me/devices`; consent é registrado por (paciente, provider, scopes) |
| Eventos cardíacos | art. 11 II "a" | Inserção crítica para alertas; mesma base que HR mas com retenção mais longa |

## 3. Princípios (skeleton)

- **Necessidade:** raw retido 90 dias; pós-90d migra para agregado diário em cold storage Parquet zstd (regra 34 + ADR 0072) — minimização agressiva
- **Finalidade restrita:** dado nunca é vendido/compartilhado; uso clínico apenas dentro do tenant
- **Cross-tenant:** apenas resumido (média 7d/30d) via passaporte ([ADR 0077](../../decisions/0077-passaporte-paciente-vinculo-cross-tenant.md)) com consent específico do paciente

## 4. Fluxo de dados (a expandir Sprint 32)

```
Provider (HealthKit/Garmin/etc) → OAuth com scopes mínimos →
  webhook/poll → Server Action wrapAction →
    INSERT device_readings (RLS, particionada mensal, regra 34) →
      job agregação diária (D+1) → device_readings_daily →
        após D+90: drop partição raw (metadata-only) → Parquet R2/S3
```

## 5. Medidas de segurança

| Camada | Controle | Referência |
|---|---|---|
| Isolamento | RLS multi-tenant + paciente owner | regras 1, 42 |
| Webhook validation | HMAC + idempotência por `external_id` | regra 33 |
| Rate limit | por provider + paciente | regra 36 |
| Retenção | raw 90d hot; agregado diário 5a; cold storage Parquet | regra 34 + ADR 0072 |
| Cross-tenant | apenas agregado via passaporte; raw nunca cruza | regra 42 |
| Direitos titular | revogação OAuth no provider + delete em LogiFit em /meu/privacidade | LGPD art. 18 |

## 6. Avaliação de risco

| Risco | Probabilidade | Impacto | Mitigação |
|---|---|---|---|
| Inferência clínica errada (alerta falso positivo) | média | médio | Alerta sempre revisado por profissional antes de notificar paciente; circuit breaker se >5%/dia false positive |
| Vazamento de séries longas reveladoras (sono/atividade noturna) | baixa | alto | Criptografia + RLS + retenção 90d raw |
| Provider externo muda termos e captura mais dado | média | médio | Auditoria semestral de scopes OAuth + termo do paciente atualizado |
| Paciente revoga OAuth mas LogiFit continua mostrando dado antigo | média | baixo | Job batch revoga ingest + offer delete em portal |

## 7. Direitos do titular

- Conectar/desconectar provider em `/app/me/devices` (paciente acessa pelo seu próprio fluxo)
- Solicitar exclusão completa em `/meu/privacidade` → cessa ingest + delete histórico (mantendo apenas agregado anonimizado para estatística do tenant, com opt-out)
- SLA **15 dias úteis**
